## 前端无人机数据不刷新问题诊断与修复

### 问题
前端页面无人机数据不再刷新，后端 API 返回 `last_seen` 停在 22 分钟前，说明 sniffer 已停止接收新数据包。

### 根因
1. **Sniffer 没有自动恢复机制**：`captureLoop` 中 `for packet := range packetSource.Packets()` 如果 channel 关闭，goroutine 直接退出，不再抓包
2. **无健康检查**：没有监控 sniffer 状态的机制，出问题后无法自动恢复

### 修复

#### 后端 (`backend/internal/drone/sniffer.go`)
- 添加 `sync.Mutex` 保护 `closed` 和 `handle` 的并发访问
- 添加 `stats.lastPacket` 时间戳，记录最后一次收到数据包的时间
- 新增 `openAndCapture()` 方法：打开网卡并启动抓包，可被重连逻辑复用
- 新增 `healthCheckLoop()`：每 10 秒检查一次，如果超过 60 秒无新数据则自动重连
- 新增 `GetLastPacketTime()` 公开方法，供 API 查询
- `captureLoop` 退出时记录 warn 日志
- `Stop()` 加锁保护

#### 后端 (`backend/internal/api/`)
- `Server` 结构体新增 `sniffer` 字段
- `NewServer()` 接受 `*drone.Sniffer` 参数
- `/health` 端点增强：返回 sniffer 状态（active、last_packet、seconds_since、stale）

#### 后端 (`backend/cmd/remoteid/main.go`)
- 传递 `sniffer` 给 `api.NewServer()`

#### 前端 (`frontend/src/App.vue`)
- 新增 `dataStaleWarning` 响应式变量
- `refreshData()` 中检测最新无人机的 `last_seen`，超过 60 秒则设为 true
- 侧边栏头部状态指示灯：正常绿色（live），数据过期红色（error）
- 数据过期时显示红色警告条："⚠ 数据已超过 60 秒未更新 — 请检查后端抓包服务是否正常运行"

### 部署说明
后端需要在树莓派上重新编译部署才能生效。

## 后端 nil slice → JSON null 全面修复 & Vite 代理/端口问题

### 问题
前端持续报 500 错误，所有 API 请求（drones、alerts）和 WebSocket 连接均失败。

### 根因
1. **Go nil slice 序列化为 JSON null**：`listDrones`、`searchDrones`、`listAlerts`、`searchAlerts` 等 handler 在无数据时返回 `null` 而非 `[]`
2. **Vite proxy 未生效**：前端通过 `127.0.0.1:8080` 直接请求后端（8080 只有 Vite dev server，没有后端），需通过 Vite proxy 转发到 `localhost:8000`
3. **Vite dev server 端口**：`vite.config.js` 中 `server.port: 8080`，代理 `/api` → `localhost:8000`，`/ws` → `ws://localhost:8000`
4. **浏览器缓存**：旧 JS bundle 可能缓存了硬编码的 API URL，需要强制刷新

### 修复
- `drones.go` `listDrones`/`searchDrones`/`getTrajectory`：nil 检查返回 `[]`
- `alerts.go` `listAlerts`/`searchAlerts`：nil 检查返回 `[]`
- `vite.config.js`：proxy target 从 `rpi5.lan:8000` 改为 `localhost:8000`
- 重启后端（8000端口）和 Vite dev server（8080端口）
- 后端 `curl localhost:8000/api/drones/` 返回 `{"drones":[],"count":0}`
- Vite proxy 验证：`curl localhost:8080/api/drones/` 返回 200
- **用户需操作**：浏览器强制刷新（Cmd+Shift+R）清除缓存

## 后端采集不到模拟无人机信号 — 修复

### 问题
用户报告后端采集不到模拟无人机信号，但其他接收软件可以看到。

### 根因
**Vite proxy 指向了 Mac 本地后端**（`localhost:8000`），而非树莓派后端（`rpi5.lan:8000`）。Mac 上没有 `wlan1` 网卡，sniffer 无法启动（被设为 nil），所以 API 返回空数据。

同时 Mac 上还跑着一个无用的 `./remoteid` 后端进程。

### 修复
1. 停掉 Mac 本地后端进程（PID 72523）
2. 将 `vite.config.js` proxy target 从 `localhost:8000` 改回 `rpi5.lan:8000`
3. 重启 Vite dev server
4. 验证：`curl localhost:8080/api/drones/` 通过 proxy 成功返回树莓派上的 2 架模拟无人机

### 当前架构
- **后端**：树莓派 `rpi5.lan` (192.168.6.220)，端口 8000，网卡 wlan2，信道 6
- **前端开发**：Mac `localhost:8080`，Vite proxy → `rpi5.lan:8000`
- **模拟器**：spoofer.py 通过串口硬件发送 Beacon 帧
- 检测到 2 架模拟无人机：`00:5b:4f:23:d8:be` (ESP32-DEMO-CCC3) 和 `02:00:00:00:00:01` (GB-OP-TEST-0001)

## 命令行数据解析工具 `ridparse`

### 新增文件
`backend/cmd/ridparse/main.go` — 独立的 Remote ID 数据解析 CLI 工具

### 功能
- **离线分析**：`ridparse -file capture.pcap` 解析 pcap/pcapng 文件
- **实时抓包**：`ridparse -iface wlan1` 实时抓包并解析
- 复用 `internal/drone/parser.go` 的 ASTM F3411 / GB42590 解析逻辑
- 复用 `internal/drone/sniffer.go` 的设备分类逻辑（OUI 检测）
- 彩色终端表格输出：MAC、标准、消息类型、UA_ID、经纬度、高度、速度、信号强度
- 无外部依赖（不需要数据库、WebSocket、API server）

### 编译
```bash
cd backend && go build -o ridparse ./cmd/ridparse/
```

### 输出示例
```

00:5b:4f:23:d8:be   GB42590    basic_id+location  Helicopter | ESP32-DEMO  23.1428700  113.2602600  45.50    2.50
```

## 一键部署脚本 `backend/deploy.sh`

采用 **rsync 源码 + 树莓派现场编译** 方案，无需 Mac 上交叉编译 CGO 工具链：
1. `rsync` 源码目录到树莓派 `/tmp/remoteid-backend/`
2. SSH 远程执行 `go build`
3. 停服 → 复制二进制和配置到 `/opt/remoteid/` → 启服
4. 首次部署时自动创建 systemd 服务

用法：`cd backend && ./deploy.sh [target_host]`（默认 `rpi5.lan`）

前置条件：树莓派需安装 Go（`sudo apt install golang`）和 libpcap-dev（`sudo apt install libpcap-dev`）

## `ridparse` 增加 TUI 监控模式

现在支持三种模式：
1. **实时抓包**：`ridparse -iface wlan1`（直接抓网卡，解析原始帧）
2. **离线分析**：`ridparse -file capture.pcap`（解析 pcap 文件）
3. **TUI 监控**：`ridparse -tui [-url http://rpi5.lan:8000]`（通过 API + WebSocket 连接后端）

### TUI 功能
- 终端全屏界面，每 2s 刷新，彩色表格展示无人机列表
- WebSocket 实时接收 `drone_update` 推送，毫秒级更新
- HTTP API 定期全量拉取（5s）兜底
- 显示 WS 连接状态（● 在线 / ● 断线）、运行时长、无人机数量
- 超过 10 秒未更新的无人机标黄 ● 提示
- 超过 30 秒无更新的无人机自动移除
- Ctrl+C 退出，自动恢复光标

## TUI 格式修复

修复了 ANSI 颜色序列导致列对齐错乱的问题：
- 改用 `padRight`/`padLeft` 函数手动控制每列宽度（不依赖 `fmt` 的 `%Ns` 格式）
- 用 `runeLenRaw` 计算纯文本宽度（排除 ANSI 转义序列）来补足 padding
- 边框宽度动态自适应列宽（`innerW` 常量）
- 列宽：MAC(18) 标准(14) UA_ID(12) 位置(19) 高度(8) 速度(8) 信号(8)
