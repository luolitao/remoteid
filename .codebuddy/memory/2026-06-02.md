## 修复 Wi-Fi Beacon 解析流程中 Message Counter 导致的无人机无法发现问题

### 根因分析

Wi-Fi Beacon Vendor Specific IE（Element ID 0xDD）中，OUI/CID 固定取值 0xFA0BBC，Vend Type 固定取值 0x0D，**Vend Type 之后还有 1 字节 Message Counter（0-255 循环计数）**，然后才是消息数据。

之前的代码：
1. `parseVendorIE()` 和 `isValidRemoteID()` 中只跳过了 OUI(3B)+VendType(1B)=4 字节，**没有跳过 Message Counter**
2. 导致 Message Counter 字节被当作消息 Header 解析
3. `parseASTMBeaconMessages()` 中取 `msgData[0] & 0x0F`（低 4 位）作为消息类型，而标准是高 4 位=消息类型、低 4 位=协议版本
4. 所有字段解析都偏移了 1 字节，解析出垃圾数据，无人机无法被发现

### 修复内容

**backend/internal/drone/parser.go**：

1. **协议概述注释**更新：明确 Vendor IE 结构包含 Message Counter 字节，Header 高 4 位=消息类型

2. **`parseVendorIE()`**：跳过 OUI(3B) + VendType(1B) + Message Counter(1B) = 5 字节
   - 循环边界 `len(raw)-4` → `len(raw)-5`
   - payload 从 `raw[idx+4:]` → `raw[idx+5:]`

3. **`parseASTMBeaconMessages()`**：消息类型从 `msgData[0] & 0x0F`（低 4 位）→ `(msgData[0] >> 4) & 0x0F`（高 4 位）

4. **`decodeASTMMessage()`**：
   - Basic ID: `idType` 和 `uaType` 的读取位置交换（高 4 位=ID Type，低 4 位=UA Type，与参考代码一致）
   - Self ID: `payload[0:23]` → `payload[1:24]`（跳过 Description Type 字节）
   - Operator ID: `payload[0:20]` → `payload[1:21]`（跳过 Operator ID Type 字节）

5. **`decodeASTMLocation()`**：
   - Direction 直接用 `float64(payload[1])`，不再 `*360/255`（与参考代码一致）
   - 长度检查 `len(payload) < 23` → `< 22`

6. **`decodeASTMSystem()`**：
   - 长度检查 `len(payload) < 23` → `< 22`
   - `flags` 变量改为直接用 `payload[0]`

7. **`parseNANAttributes()`**：NAN Vendor Specific Attribute 中也跳过 Message Counter（`attrValue[4:]` → `attrValue[5:]`，长度检查 `>= 4` → `>= 5`）

**backend/internal/drone/sniffer.go**：

8. **`isValidRemoteID()`**：跳过 OUI(3B) + VendType(1B) + Message Counter(1B) = 5 字节
   - 循环边界 `len(raw)-4` → `len(raw)-5`
   - payload 从 `raw[i+4:]` → `raw[i+5:]`
   - Header 检查条件调整：`firstNibble <= 5 && lowNibble == 2`（高 4 位=消息类型 0-5，低 4 位=协议版本 2）

编译验证通过。

## 无人机解析"无有效数据"问题修复

### 根因
通过 hex dump 发现：OUI(FA:0B:BC) + VendType(0x0D) 之后，除了 Message Counter(1B) 还有额外的元数据字节（如 `F2 19 05`），导致固定偏移 5 字节后取到的不是 ASTM 消息 Header（0xF2 高 4 位=15，不是合法的消息类型 0-5）。

### 修复方案
不再假设 OUI+Type 之后固定偏移，改为**扫描查找** ASTM 消息 Header。

**parser.go**：
1. **`parseVendorIE()`**：OUI+Type 匹配后，调用 `findASTMMessageHeader()` 扫描最多 8 字节找 Header
2. **`parseNANAttributes()`**：同样使用 `findASTMMessageHeader()` 扫描
3. **新增 `findASTMMessageHeader()`**：在 OUI+Type 后扫描，找满足条件的字节：高 4 位=消息类型(0-5)，低 4 位=协议版本(2)，最多扫描 8 字节

**sniffer.go**：
4. **`isValidRemoteID()`**：同样改为扫描方式（OUI+Type 后扫描最多 8 字节），不再假设固定 5 字节偏移

### 日志时间格式优化
- **`cmd/remoteid/main.go`** init()：`ReplaceAttr` 自定义时间格式为 `2006-01-02T15:04:05.000`（毫秒精度，无时区）
- **`internal/config/config.go`** SetLogLevel()：同步使用相同 ReplaceAttr，同时输出到 `os.Stdout`（原来错误地用了 `os.Stderr`）

编译验证通过。

## 运行时输出协议管理信息（频段/传输类型/OUI）

### 需求
运行时输出接收协议信息：WiFi 2.4G / 5G 频段、Beacon / NAN 传输类型、Beacon OUI 等。

### 修改文件

**backend/internal/drone/sniffer.go**：

1. **`processPacket()`**：从 RadioTap 层提取 `ChannelFrequency`（`uint16(radioTap.ChannelFrequency)` 类型转换），用于判断 2.4G/5G 频段
2. **`getBandLabel()`** 新增函数：根据频率返回 `WiFi 2.4G(2412MHz)` / `WiFi 5G(5180MHz)` / `WiFi(Unknown)`
3. **`classifyDeviceWithOUI()`** 新增方法：设备分类同时返回检测到的 OUI 标签
4. **`detectBeaconOUI()`** 新增函数：返回 `FA:0B:BC(ASD-STAN)` / `06:05:04(LegacyASTM)` / `""`
5. **日志格式更新**：`slog.Info("检测到真实无人机", "mac", ..., "band", ..., "transport", ..., "oui", ..., "signal_dbm", ...)`

**backend/cmd/ridparse/main.go**：

6. **`isValidRemoteID()`** 修复：同样跳过 Message Counter（与 sniffer.go 保持一致，`len(raw)-5`，`raw[i+5:]`）
7. **`processPacket()`**：提取频段/传输/OUI，传递给 `printDroneInfo`
8. **`detectOUI()`** 新增函数：返回 OUI 标签
9. **`printHeader()`** 和 **`printDroneInfo()`**：增加"传输"（Beacon/NAN）和"频段"列，OUI 信息单独一行输出

编译验证通过。
