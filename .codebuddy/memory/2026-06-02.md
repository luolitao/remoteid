# 2026-06-02

## 修复 Wi-Fi Beacon 解析流程中 Message Counter 导致的无人机无法发现问题

### 根因分析

Wi-Fi Beacon Vendor Specific IE（Element ID 0xDD）中，OUI/CID 固定取值 0xFA0BBC，Vend Type 固定取值 0x0D，**Vend Type 之后还有 1 字节 Message Counter（0-255 循环计数）**，然后才是消息数据。

之前的代码：
1. `parseVendorIE()` 和 `isValidRemoteID()` 中只跳过了 OUI(3B)+VendType(1B)=4 字节，**没有跳过 Message Counter**
2. 导致 Message Counter 字节被当作消息 Header 解析
3. `parseASTMBeaconMessages()` 中取 `msgData[0] & 0x0F`（低 4 位）作为消息类型，而标准是高 4 位=消息类型、低 4 位=协议版本
4. 所有字段解析都偏移了 1 字节，解析出垃圾数据，无人机无法被发现

### 修复内容

**backend/internal/drone/parser.go**：
1. 协议概述注释更新，明确 Vendor IE 结构包含 Message Counter 字节
2. `parseVendorIE()`：跳过 OUI(3B) + VendType(1B) + Message Counter(1B) = 5 字节
3. `parseASTMBeaconMessages()`：消息类型从低 4 位改为高 4 位
4. `decodeASTMMessage()`：Basic ID 的 id_type/ua_type 交换位置，Self ID/Operator ID 跳过 type 字节
5. `decodeASTMLocation()`：Direction 直接用 uint8，不再 *360/255
6. `decodeASTMSystem()`：长度检查修正
7. `parseNANAttributes()`：NAN Vendor Specific Attribute 中也跳过 Message Counter

**backend/internal/drone/sniffer.go**：
8. `isValidRemoteID()`：跳过 Message Counter，Header 检查条件调整

编译验证通过。

## 无人机解析"无有效数据"问题修复

### 根因
通过 hex dump 发现：OUI(FA:0B:BC) + VendType(0x0D) 之后，除了 Message Counter(1B) 还有额外的元数据字节（如 `F2 19 05`），导致固定偏移 5 字节后取到的不是 ASTM 消息 Header（0xF2 高 4 位=15，不是合法的消息类型 0-5）。

### 修复方案
不再假设 OUI+Type 之后固定偏移，改为**扫描查找** ASTM 消息 Header。

**parser.go**：
1. `parseVendorIE()`：OUI+Type 匹配后，调用 `findASTMMessageHeader()` 扫描最多 8 字节找 Header
2. `parseNANAttributes()`：同样使用 `findASTMMessageHeader()` 扫描
3. 新增 `findASTMMessageHeader()`：扫描满足高 4 位=消息类型(0-5)，低 4 位=协议版本(2) 的字节

**sniffer.go**：
4. `isValidRemoteID()`：同样改为扫描方式（OUI+Type 后扫描最多 8 字节）

### 日志时间格式优化
- `cmd/remoteid/main.go` init()：`ReplaceAttr` 自定义时间格式为 `2006-01-02T15:04:05.000`（毫秒精度，无时区）
- `internal/config/config.go` SetLogLevel()：同步使用相同 ReplaceAttr，输出到 os.Stdout

编译验证通过。

## 重新引入 GB 42590-2023 国标协议解析，修复国标模拟器数据解析错误

### 问题根因

国标模拟器发出的数据使用 `FA:0B:BC` + `0x0D` OUI（与 ASTM 相同），但 Header 字节低 4 位协议版本为 `0x1`（ASTM 为 `0x2`）。之前的代码移除了 GB 解析逻辑后，国标数据走了 ASTM 解析路径，导致：

1. **UAS ID 和 Operator ID 错误对换**：ASTM 路径解析 GB 数据时字段可能错位
2. **坐标错误**：GB 方向编码为 12 位（高 4 + 低 8），ASTM 为 8 位 + EW 标志，解析错位导致坐标偏移
3. **高度 8000-10000 米**：字段偏移错位后 altitude 读到非高度字段，产生荒谬值

### 修复内容

**`backend/internal/drone/parser.go`**：
1. 常量新增 `gbProtocolVersion = 1`
2. `findASTMMessageHeader()` 同时识别 GB (protoVer=1) 和 ASTM (protoVer=2)
3. `parseASTMBeaconMessages()` 根据 protoVer 分派到 `decodeGBMessage()` 或 `decodeASTMMessage()`
4. 新增 GB 解码函数：`decodeGBMessage()`, `decodeGBLocation()`, `decodeGBSystem()` 及辅助函数
5. GB Location 方向编码使用 12 位 * 360/65535，带无效值检测

**`backend/internal/drone/sniffer.go`**：`isValidRemoteID()` 支持 protoVer=1

**`backend/cmd/ridparse/main.go`**：`isValidRemoteID()` 支持 protoVer=1

**`backend/internal/drone/processor.go`**：高度合理性检查（>5000m 或 <-500m 清零）

编译验证通过，无新增 lint 错误。
